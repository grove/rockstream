## Notes

- `ShardManager` is an in-memory store backed by `Arc<RwLock<_>>` — it is not
  persisted to disk in v0.29. Persistence and SSI-backed lease transactions
  are planned for v0.31 (frontier protocol), consistent with the ROADMAP scope.
- Per-shard SlateDB handle creation (opening one `ShardDb` per lease) is wired
  at the `ShardManager` / `ShardScheduler` layer but the actual `Db::open` call
  is deferred to when a worker executes its first operator task on the shard;
  this avoids eagerly opening databases for shards that have not yet received
  any data. The `ShardDbBuilder` from `rockstream-storage` is the correct
  integration point.
- The `ShardScheduler::on_worker_dead` path is exercised in unit tests via
  direct method calls. The full async "TCP drop → release" path is separately
  covered by the `shard_leases_released_on_worker_disconnect` integration test.
