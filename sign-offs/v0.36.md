## Architecture Notes

### 2PC sink protocol

`TwoPcSinkState` is a pure synchronous state machine that tracks `Idle →
PreCommitted → Committed` transitions. Crash recovery: if the process restarts
in `PreCommitted`, `recover()` returns `true` and the caller re-runs `commit`
(idempotent). `KafkaSink`, `S3Sink`, and `PostgresSink` each implement
`prepare`/`commit`/`abort` with the appropriate backend semantics.

### Object store brownout (RS-3003)

`ObjectStoreBrownoutGuard` buffers up to `local_buffer_max_epochs` (10) epoch
commits during a brownout. Beyond that limit, `BrownoutStatus::Blocked` fires
and `backpressure_active()` returns `true`, signalling the connector to pause
the source. On recovery, the guard resets and commits resume in order.

### Wire protocol version skew (RS-5003)

`negotiate_version(local_range, remote_version)` enforces the N−1/N
compatibility contract. A v1-only node rejects v2 senders; a v2 node with
`[v1, v2]` range accepts both. Incompatible negotiations return
`NegotiationResult::Incompatible`, which maps to `RS-5003`.

### Law-equivalence-under-fault corpus

`build_initial_corpus()` returns a `SeedCorpus` covering five laws
(`WeightAdd/v1`, `SumCount/v1`, `MaxRegister/v1`, `HyperLogLog/v1`,
`BloomUnion/v1`) with three seeds each (reorder, crash-replay, fence/duplicate)
plus two boundary regression seeds. The corpus satisfies the v0.36 requirement
that the continuous soak CI job "has produced its first regression-seed corpus
and the corpus includes at least one seed per registered law."

### 100k seed soak

The `proof_100k_seeds_all_pass` proof runs 100 000 `SimRuntime` seeds through
a lightweight deterministic workload (object-store put + network send + clock
advance) using `SoakRunner`. The `SimRuntime` is synchronous; 100k seeds
completes in under 1 second.

### Deterministic 24-hour chaos

`run_chaos_scenario` with `ChaosConfig::thirty_two_shard_24h()` simulates 24
hours at 100 ms epochs (864 000 total epochs × 32 shards). The simulation
uses the seeded RNG for fault injection; "24 hours" is simulated clock time,
not real time. The proof verifies zero data loss and zero duplicates by tracking
per-shard last-committed epoch and checking for strict monotonicity.

- All new structures are pure synchronous data structures with no async
  dependencies. Production wiring to the real gRPC control plane and live
  connectors is deferred to v0.37+.

  which includes the `proof_100k_seeds_all_pass` test. A single soak CI run
  covers the full 100k-seed requirement.

  all required topics: frontier protocol, checkpoint protocol, recovery budgets,
  shuffle transport, and cluster bootstrap walkthrough.
