## Architecture Notes

### Barrier injection

`CheckpointBarrier` is stamped with a sequential `CheckpointId` and a `barrier_epoch`.
The barrier flows through the data path; when an operator receives it, it drains
in-flight state and sends a `ShardCheckpointAck` back to the coordinator.

### Bounded alignment buffers

Multi-input operators (joins, unions) may receive the barrier on one input before
the other.  `AlignmentBuffer` holds fast-side rows with a hard `max_rows` limit.
When the limit is reached, `push()` returns `RS-1601` — the buffer never grows
unboundedly.  This is the key invariant that satisfies the "never grows unbounded"
exit criterion.

### Atomic cluster checkpoint commit

`CheckpointCoordinator` collects `ShardCheckpointAck` messages.  All registered
shards are pre-loaded into a `pending: HashSet<ShardId>` at barrier injection time.
`Committed` is only returned when the set is empty — i.e., every shard has
acknowledged.  `abort()` immediately transitions to `Recovering`/`RS-1602` and
clears the in-progress state so a new barrier can be injected after recovery.

Only one checkpoint may be in-flight at a time; attempting to inject a second
barrier while one is in progress returns `RS-1602`.

### Old checkpoint GC

`CheckpointGc` (embedded in `CheckpointCoordinator`) tracks `(CheckpointId, barrier_epoch)`
pairs.  `gc_old_checkpoints(frontier_epoch)` returns IDs whose `barrier_epoch <
frontier_epoch`; those checkpoints are safe to delete from durable storage.
The caller issues the actual storage `delete` calls, keeping the GC logic pure
and testable without object-store mocks.

### Error codes

- `RS-1601`: Checkpoint alignment buffer overflow. Bounded; returns an error rather than growing.
- `RS-1602`: Cluster checkpoint RECOVERING state. Surfaced on abort or double-inject attempt.

## Notes

- This version implements the cluster checkpoint protocol as an in-process simulation.
  Production wiring to SlateDB snapshot handles and actual object-store writes will
  be added in v0.35+ alongside the recovery driver.
- `RS-1601` and `RS-1602` are in the `1600`-range (checkpoint/recovery block).
- The `AlignmentBuffer` and `CheckpointGc` are pure data structures with no async
  dependencies, making them suitable for unit testing without any runtime setup.
