## Architecture Notes

### Split state machine

`ShardSplitOp` tracks the full lifecycle from checkpoint injection through
donor cleanup.  The `split_key_hash` is the midpoint of the donor's hash range:
keys with `hash(key) < split_key_hash` remain on the donor; keys with
`hash(key) >= split_key_hash` move to the new shard.  The scan-and-delete
cleanup pattern (no range delete) is enforced by `MigrationStats.rows_migrated`
matching `rows_to_cleanup` at `Done` transition.

### TombstoneGc compaction

`TombstoneGc` is invoked by the compaction filter when a value equals the law's
identity element (`LawBundle::is_identity`).  After split cutover, the donor's
migrated key range is cleaned up by scanning and deleting entries —
`simulate_split_migration` in `compaction.rs` proves this contract
synchronously without requiring SlateDB or a real compaction trigger.

### OR-Set/v1 law

`OrSet/v1` is a semilattice merge law for causal OR-Set arrangements.  Wire
format: `[count: u32 BE] [element_id: u64 BE, tag: u64 BE] × count`.  Merge
is set-union (sort + dedup).  The identity element is the empty set (count=0).
Full user-visible `OR_SET` column types and tombstone-delete semantics ship in
v0.44.

### Simulation scope

All v0.37 proofs are synchronous unit/integration tests with no async
dependencies.  The production shard-split execution engine (checkpoint barrier
injection, gRPC control-plane split coordinator, live epoch cutover) is wired
in v0.38 when the proactive scaling machinery is built.
