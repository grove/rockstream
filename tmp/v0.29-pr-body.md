## Summary

Implements **v0.29.0: Shard leasing and scheduling** â€” the second increment of Phase 4 (Distributed Core).

This version adds a monotone-fencing-token lease system that ensures only one worker can commit writes to any given shard at any time, plus a scheduler that re-assigns shards automatically when a worker dies.

## What is new

### `rockstream-types`: Shard lease types

- `lease.rs` (new): `ShardLease` (shard_id, worker_id, lease_token, acquired_at_ms) and `ShardRevokeReason`.
- `ControlMessage` extended: `ShardAssigned`, `ShardRevoked`, `FenceAck`.
- `WorkerMessage` extended: `RequestShard`, `FenceWrite`.

### `rockstream-control`: `ShardManager`

Thread-safe in-memory shard lease store with monotonically increasing fencing tokens.

- `acquire(shard_id, worker_id)`: issues a fresh token; same worker can reacquire; different worker gets `RS-1601`.
- `force_acquire`: evicts any current holder for rebalancing.
- `release_worker(worker_id)`: atomically removes all leases for a dead worker.
- `is_valid_writer(shard_id, token)`: the write fence â€” returns false the instant a newer token is issued.
- Global token counter means tokens are monotone across ALL shards and ALL workers.

### `rockstream-control`: `ShardScheduler`

Combines `TopologyCatalog` + `ShardManager` + `PlacementAlgorithm`.

- `assign_initial_shards`: distributes shards to best-headroom workers; skips shards already held by healthy workers.
- `on_worker_dead`: releases the dead worker's leases, reassigns each shard to the next best healthy worker with a strictly higher token.

### `rockstream-control`: `ControlService` extended

- `RequestShard` message: acquire a lease and reply `ShardAssigned`.
- `FenceWrite` message: check token validity and reply `FenceAck`.
- `Deregister`: release all leases, send `ShardRevoked` per freed shard, then `TopologyChanged`.
- TCP disconnect (no Deregister): `release_worker` cleans up all leases automatically.

## Notes

- `ShardManager` is in-memory only in v0.29. Persistence and SSI-backed lease transactions are planned for v0.31.
- Per-shard `ShardDb` opening is wired at the scheduler layer but actual `Db::open` is deferred to first operator task execution, consistent with the existing `ShardDbBuilder` integration pattern.
