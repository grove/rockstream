## Summary

Implements v0.23.0: **Bootstrap and snapshot mode**.

Adds `PlanNode::Snapshot` and a `SnapshotOp` physical operator that delivers an existing relation as a sequence of insert-only bootstrap epochs. Once all rows have been emitted, `is_complete()` signals the bootstrap frontier. `resume_from(N)` enables reconciliation after connector position loss by skipping already-committed rows.

## Changes

### Plan IR
- `PlanNode::Snapshot { source_name, batch_size }`
- `OpKind::Snapshot { source_name, batch_size }`

### Differentiator
- `Snapshot` → `Stateless` (insert-only source; no arrangement)

### Physical operator (`rockstream-ops`)
- `SnapshotOp`: streamed bootstrap with `next_batch()`, `drain_all()`, `resume_from(N)`
- `BootstrapPhase` enum: `Bootstrapping` / `Complete`
- `is_complete()`: bootstrap frontier signal
- `rows_delivered()`, `total_rows()`, `rows_remaining()`: watermark tracking
- `RS-1010` returned by `resume_from` when watermark exceeds snapshot size

### Oracle / tests (`rockstream-oracle`)
- `BootstrapOracle`: batch reference with `batches()`, `merge_all()`, `resume()`
- `sorted_rows()` helper for deterministic comparison
- 11 proof tests: delivery completeness, oracle equivalence, resume no-gaps, resume no-duplicates, `is_complete` lifecycle, plan codec round-trip, diff law, explain label, 100k-row scale, idempotent replay, connector position-loss reconciliation

### Supporting changes
- `RS_1010` error code registered
- Catalog codec encode/decode for `PlanNode::Snapshot`
- Explain labels in plan and runtime crates
- `op_kinds` / `assert_op_kinds` updated in SQL frontend
- Version bumped `0.22.0` → `0.23.0`
- `sign-offs/v0.23.md` added
