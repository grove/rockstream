v0.28.0: Control plane and worker discovery

Implements Phase 4 entry: control service, worker registration, topology
catalog, placement algorithm, mTLS scaffolding, and Tier 1/2 role flows.

## What's new

### rockstream-types
- WorkerId strong-typed ID added to ids.rs
- New topology module: NodeRole, CapacityHeadroom, WorkerRegistration,
  WorkerInfo, ControlMessage, WorkerMessage

### rockstream-control
- TopologyCatalog: thread-safe in-memory worker registry (register,
  heartbeat, deregister, healthy_workers, versioned mutations)
- PlacementAlgorithm: capacity-headroom-aware choose and assign_n;
  tie-breaking by lowest WorkerId for determinism
- ControlService + ControlServiceHandle: TCP server accepting
  newline-delimited JSON WorkerMessage frames; audit-log integration
- TlsConfig + LoadedTls: mTLS scaffolding; validates PEM files at startup

### rockstream (binary)
- --role=control Tier 1: starts ControlService on --control-bind
- --role=worker Tier 1: connects to --control=<addr>, registers, runs
- --role=all Tier 2: starts control service and self-registers in one process
- bootstrap subcommand: probes a control service and prints readiness
- --tls-cert / --tls-key / --tls-ca-cert flags for mTLS on all channels

## Proof criteria verified
- Tier 1 and Tier 2 start flows: integration tests in service.rs
- Workers join through --control=<url> and appear in topology catalog
- Topology changes are audited (worker.registered, worker.deregistered)
- Placement algorithm respects capacity_headroom (unit + property tests)
- mTLS scaffolding compiles and validates PEM files at startup
